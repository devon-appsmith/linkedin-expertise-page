Go / Golang

Go and the discipline of simplicity

✦ 23 posts synthesized

Updated March 2026

I came to Go in 2018, skeptical. At the time I was deep in a Java microservices architecture that had accumulated layers of abstraction — factories that created factories, configuration objects that wrapped configuration objects. When a teammate pushed for rewriting our event-ingestion pipeline in Go, I was the one asking "but what about generics? What about inheritance?" What I didn't appreciate yet was that those absences were the point. Go wasn't incomplete. It was disciplined.

Six weeks into our Go rewrite and I finally get it. I spent the first two weeks fighting the compiler and missing my IDE's autocomplete. Now I'm reading other people's Go code and understanding it instantly. When was the last time I could do that with a Java service I didn't write? The readability is the killer feature — and I don't think it's an accident.

Go's approach to error handling was the second thing that changed my thinking. "Just check the error" sounds patronizing until you've traced a production incident through five layers of exception wrapping to find the original cause was a connection timeout. Explicit error values force you to think at every call site: what do I do if this fails? That's not boilerplate — that's the actual engineering work, made visible. I've come to see callers that panic on errors as a smell: they're encoding an assumption that was never examined.

Goroutines and channels took longer for me to internalize correctly. My early Go code used channels everywhere because I'd heard "don't communicate by sharing memory; share memory by communicating" and took it too literally. In practice, channels are for synchronization and sequencing. For simple shared state, a mutex is often clearer and measurably faster. The lesson was about reading idioms carefully rather than applying them as blanket rules.

Hot take: most Go concurrency bugs I've seen in code review aren't from misusing goroutines — they're from engineers who reached for a channel when they wanted a mutex. "Share by communicating" doesn't mean channels are always the answer. Know your primitives. Channels are queues, not magic concurrency dust. If you don't need a queue, you probably don't need a channel.

Where I've landed after seven years of writing Go professionally: the language is most powerful when you resist the urge to abstract too early. Interfaces in Go are implicit, which means you should only define them when you genuinely need them — usually at package boundaries, almost never inside a single package. I've reviewed codebases where every struct has a corresponding interface, and the result looks Java-shaped: a pyramid of indirection that makes testing feel clean but reasoning feel hard. The idiomatic path is to write concrete types first, let patterns emerge, and extract interfaces when you feel the pull of duplication or the need for a testable seam.

Code review comment I keep leaving on Go PRs: "Does this interface have more than one implementation?" If no, delete the interface and use the concrete type. In Go, interfaces are discovered, not designed upfront. An interface with one implementation isn't abstraction — it's indirection with extra ceremony. And indirection has a cost that compounds across a codebase.

Go 1.18's generics were a moment I watched closely. I'd argued for years that the language was fine without them — that the constraint forced cleaner thinking. I still believe the pre-generics discipline was underrated. But I was wrong to be a generics skeptic outright. There's a real class of problems — typed collections, transform pipelines, result types — where code duplication was the only alternative, and duplication is its own complexity tax. The implementation landed thoughtfully. I now use generics sparingly, mostly at the edges of library-facing code, and I remain skeptical of heavy generic abstractions inside application logic. The discipline doesn't disappear with generics; it just moves.

Five years of production Go across three different teams. What nobody warns you about: the hardest part isn't the language — it's that Go makes all your bad architectural decisions very easy to see. There's nowhere to hide complexity. That's uncomfortable at first, and then it's exactly what you wanted all along.
