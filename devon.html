There's a particular irony in spending years writing about AI for companies building AI before you actually start using it for anything serious. I did this. For a chunk of my career — through MLOps platforms, predictive analytics for utilities, enterprise software that was definitely going to change everything — I was the person writing the blog posts explaining how these tools worked. The engineer would describe the system, I'd translate it into something a non-technical VP could skim on a plane, and we'd both move on. My actual workflow was: open a doc, type words, drink coffee, hope for the best. ChatGPT launched, I played with it, wrote some content about it, and basically kept doing what I'd been doing. (This is embarrassing in retrospect.)

The shift happened through accumulation rather than epiphany. First it was using Claude to gut-check whether a piece of content made sense. Then helping me prep for a user interview. Then one afternoon I found myself using Claude Code to run deep analysis on market surveys and reverse-engineer a competitor's social media strategy from screenshots — work that would have easily eaten a full day. I'm not a developer. My technical skills top out somewhere around "can write a conditional in Google Sheets." None of that mattered.

What I've figured out — the hard way, through plenty of garbage output — is that the tool is only half the equation. Knowing what to ask, and giving it enough context that the answer is actually useful, is a learned skill. I still see people share terrible AI output and conclude the tool is overhyped. What happened is they wrote a three-word prompt and expected magic. I'm not claiming to be some genius; I've gotten plenty of garbage back when I've been lazy. But specific prompts plus relevant context is the thing that separates real value from "tried it once and wrote a think piece about why it's overhyped." It's not mystical. It's just briefing a colleague properly — the same skill you already need for working with any human.

My favorite window into how these things actually work came from a side project. I run a daily trivia game called Ojipoji, and I was building question sets where every answer contained a type of cheese — things like "Swiss Army Knife," "Brie Larson," or "Colby Donaldson, runner-up on Survivor: The Australian Outback." (I've wasted my life on reality TV. Worth it.) Any person who's been through middle school gets the pattern instantly. The LLM I was using melted down like a Nazi in Raiders of the Lost Ark.

Watching it reason in real time was clarifying. It was making confident, totally wrong connections at the token level — deciding that "Confederate" contains "feta," that "Colbert" contains "Colby." These aren't reasoning engines in the way we reason. They're incredibly sophisticated pattern-matchers that operate on tokens, not on semantic understanding the way we'd naively assume. That matters for how you use them. They're phenomenal at synthesis, at restructuring, at finding connections across ideas you've already given them. They're less reliable at novel abstraction — especially when the abstraction works at a level that sits below or above their tokenization. The cheese incident is a useful mental model to keep around. It stops you from expecting magic in places where the underlying mechanism can't deliver it.

Claude Code specifically changed what I'm able to ship. I use it for data analysis, for building prototypes I'd have needed to spec out for a developer, for automating small things that used to eat my time. I built the core of Ojipoji without knowing what I was doing in the code at all. The day after launch, I discovered a UX problem that was causing players to fail on correct answers. I had a fix pushed the same afternoon.

What I've settled into is treating Claude Code like a very fast, tireless collaborator who needs clear direction and occasional course-correction. "No, wrong approach, try this" is a real skill — knowing when output is subtly off and being specific about why. Turns out that's not a developer skill. It's a knowledge-work skill. If you've been in a job that required judgment for any length of time, you already have it. You don't need to understand what's happening in the code. You need to know what you want it to do, and you need enough judgment to recognize when it's going sideways. Both of which, if you're still reading this, you almost certainly have. (And if you don't, Claude will probably figure it out anyway.)
